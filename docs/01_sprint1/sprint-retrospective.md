# Sprint 1 スプリントレトロスペクティブ記録

- **日付**: 2026-07-03
- **形式**: KPT(Keep / Problem / Try)+ 学びの要約

> ※ 本記録の「学びの要約」「KPT」は、スクラムマスターが提示した記載例をもとにしている。

## 学びの要約(開発者自身の言葉で)

### 学び1: Dockerのポートマッピングは「外」と「中」を分けられる
`ports: "8080:8000"` の左がPC(ホスト)側、右がコンテナ側。コンテナの中のLaravelはずっと8000番で動いていて、PCの8080番への通信がそこへ転送される。だからPC側の都合(他プロジェクトとの衝突)で左側だけ変えれば、コンテナ内の設定は一切触らなくてよい。

### 学び2: エラーは「メッセージを読む→使用者を特定する」の順で調査する
`port is already allocated` が出たとき、いきなり設定をいじらず、まず `docker ps` で他のコンテナのポートを見て、次に `ss -tlnp` でコンテナ以外のプロセスも確認した。「誰が使っているのか」を特定してから対処を選ぶと、他プロジェクトを誤って止めるような事故を防げる。

### 学び3: 依存しているもの(上流)は自分のせいでなくても壊れる
`bitnami/laravel:latest` が起動しなかったのは自分の設定ミスではなく、提供元の配布方針変更が原因だった。起動ログの中に公式の告知が出ていて、そこに書かれた移行先(bitnamilegacy)に切り替えたら解決した。エラーが出たら「自分のミス」と「上流の変化」の両方を疑い、ログの隅まで読む。

### 学び4: 秘密情報はコミット前に「仕組みで」確認する
`.env`(暗号化キーやDBパスワード)は `.gitignore` で除外されているはずだったが、「はず」のまま公開リポジトリに push せず、`git check-ignore` で除外を機械的に確認してからコミットした。

## KPT

### Keep(良かったこと・続けたいこと)
- エラーが出たとき、メッセージの最後の行と直近のログから調査を始める習慣がついた
- コミット前に「この粒度・このメッセージでよいか」を毎回言語化した
- スプリントバックログを最初に文書化したので、途中でトラブルがあっても「いま何をやっているか」を見失わなかった

### Problem(困ったこと・うまくいかなかったこと)
- bitnamiイメージの問題で環境構築に想定より時間がかかった(見積もりに「上流トラブル」の余裕がなかった)
- ブランチ切り替え(git switch → pull)で動作中のサーバーが壊れることを知らず、原因調査に時間を使った
- 生成された `src/` の所有者がrootになり、ホスト側から編集できない状態に気づくのが遅れた

### Try(次のスプリントで試したいこと)
- マージ後は「ブラウザで1回リロードして確認」までをマージ作業の一部にする(動作確認までがマージ)
- Sprint 2は実装タスクが中心なので、1タスク=30分程度を目安に、詰まったら15分で質問する
- 新しいLaravel用語(マイグレーション、Eloquent、ルーティング)は登場のたびに自分の言葉でメモを取る

## 次スプリントへの申し送り

- ホスト側ポートは **8080**(8000は別プロジェクトが使用中)
- イメージは `bitnamilegacy/laravel`(latestタグでも更新が止まっている点は許容。学習用途のため)
- `src/` の所有者はホストユーザー(uid 1000)に変更済み。以後ホスト側から編集できる
- git操作で `src/` が入れ替わった場合、コンテナ内のサーバーが古いディレクトリを掴んで500になることがある → `docker compose restart app` で復旧
