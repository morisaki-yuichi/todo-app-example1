## 2. 質疑応答と決定事項

### Q1. TODO の項目(カラム)はどこまで持たせるか?

**決定: 最小構成**

| 項目 | 型 | 説明 |
|---|---|---|
| title | 文字列 | TODO のタイトル |
| description | テキスト(NULL 可) | 内容・詳細メモ |
| completed | 真偽値 | 完了状態(初期値 false) |

- 理由: 4 スプリントで確実に CRUD を完了させることを優先。期限日・優先度は今回のスコープ外とする。
- ※ このほか Laravel の規約により `id` `created_at` `updated_at` が自動で付く。

### Q2. バリデーションとソート順は?

**決定: 以下の推奨案を採用**

- **バリデーション**
  - タイトル: 必須・最大 100 文字
  - 内容: 任意・最大 1000 文字
  - 完了状態: true / false のみ
- **一覧のソート順**: 作成日時の新しい順(降順)
- **エラー表示**: 入力フォームの上に日本語で表示し、入力値は消さずに残す(old 値)

### Q3. 認証機能(ログイン)は含めるか?

**決定: スコープ外**

- 認証なしの単一ユーザーアプリとする。
- 理由: 今回の学習テーマは CRUD の習得。認証は次のプロジェクトの学習テーマとする。

### Q4. 1 スプリントあたりの想定作業時間は?

**決定: 3〜4 時間**

- タスク量(ベロシティ)の見積もりはこれを基準にする。

### Q5. GitHub リポジトリの運用は?

**決定: パブリックリポジトリを新規作成**

- `morisaki-yuichi` アカウント配下に公開リポジトリとして作成し、スプリントごとにプッシュする。

### Q6. ブランチ戦略は?

**決定: GitHub Flow**

- `main` は常に動く状態を保つ。
- 作業は `feature/xxx` ブランチで行い、完成したら `main` にマージする。
- コミットは Conventional Commits(`feat:` `fix:` `docs:` `chore:` など)に従う。
- 粒度は「1 コミット = 1 つの意味のある変更」。動かない中途半端な状態ではコミットしない。
