# Laravel 概念解説集 〜このリポジトリを理解するための前提知識〜

[開発トレースガイド](dev-walkthrough.md) でコードの差分を読む前に、
登場する概念をここで押さえられるようにしたもの。**開発中に実際に説明された内容**を、
「一言定義 → なぜ必要か → このリポジトリでの実例」の形で登場順にまとめている。

## 前提編: Docker と HTTP

### コンテナ / イメージ / Docker Compose
- **定義**: コンテナ = アプリを動かす「隔離された小さな実行環境」。イメージ = コンテナの型(テンプレート)。Docker Compose = 複数コンテナをまとめて定義・起動する仕組み。
- **なぜ**: 自分の PC を汚さずに Laravel や MySQL を動かせる。「PC に直接インストールしたら他プロジェクトと競合した」という事故を構造的に防ぐ。
- **実例**: `docker-compose.yml` — app(Laravel)と mysql の 2 コンテナ構成。[Step 2](dev-walkthrough.md#step-2-docker-compose-定義--884b072)

### ポートマッピング
- **定義**: `ports: "8080:8000"` の左が PC(ホスト)側、右がコンテナ側。PC の 8080 番への通信がコンテナの 8000 番へ転送される。
- **なぜ**: 外(PC)と中(コンテナ)で番号を分けられるので、PC 側の都合(他プロジェクトとの衝突)が起きても**コンテナ内は無変更**で対処できる。
- **実例**: 開発中に実際にポート 8000 が別プロジェクトと衝突し、左側だけ 8080 に変更した([Sprint 1 レビュー](../01_sprint1/sprint-review.md) 問題 1)。

### ボリューム(データの永続化)
- **定義**: コンテナ内のディレクトリを PC 側のフォルダや名前つき領域と同期する仕組み。
- **なぜ**: **コンテナは消えるとデータも消える**。コード(`./src:/app`)や DB データ(`mysql_data:`)は外に置くのが鉄則。
- **実例**: `docker-compose.yml` の `volumes:` 2 箇所。

### HTTP メソッドの使い分け(最重要原則)
- **定義**: GET = 「見るだけ」。POST/PUT/PATCH/DELETE = データを変える操作(順に: 作成/全体更新/一部変更/削除)。
- **なぜ**: **GET でデータを変えてはいけない**。ブラウザの先読みや検索エンジンのクローラはリンク(GET)を勝手に辿るので、たとえば完了切替を GET リンクにすると「勝手に TODO が完了になる」事故が起きうる。
- **実例**: 完了切替は PATCH フォーム(`todos/index.blade.php`)、削除は DELETE フォーム(`todos/delete.blade.php`)。[Step 11](dev-walkthrough.md#step-11-完了切替us-5--5206d3c)

### HTTP ステータスコード(このプロジェクトに登場したもの)
- **200** 成功 / **302** リダイレクト(別 URL へ案内)/ **404** 見つからない / **419** CSRF トークン切れ・不足(Laravel 特有)/ **500** サーバー内部エラー。
- **なぜ知るか**: エラー調査の第一手がかり。Chrome の「This page isn't working」は 500 系の画面、など**ブラウザ表示とコードを対応づけて読む**。

## Laravel 編(Sprint 2 で登場)

### MVC
- **定義**: アプリを 3 つの役割に分担する設計。**M**odel = DB とのやりとり担当、**V**iew = 画面(HTML)担当、**C**ontroller = 司令塔(リクエストを受け、モデルに仕事を頼み、結果をビューに渡す)。
- **リクエストの流れ(一本道)**: `URL(ルーティング)→ コントローラ → モデル ⇔ DB → ビュー → HTML`
- **なぜ**: 役割が分かれていると「どこを直せばいいか」「どこで止まったか」を絞り込める。エラー調査もこの一本道のどこで止まったかを考える。
- **実例**: [Step 5](dev-walkthrough.md#step-5-一覧表示us-2--edd271d) の差分(`edd271d`)が最小構成の教科書。

### ルーティング(routes/web.php)
- **定義**: 「この URL・この HTTP メソッドで来たら、このコントローラのこのメソッドを呼ぶ」という対応表。
- **名前つきルート**: `->name('todos.index')` と名付けておくと、ビューでは `route('todos.index')` と書ける。将来 URL を変えてもビューの修正が不要。
- **実例**: `src/routes/web.php`。全ルートは `php artisan route:list` で一覧できる。

### artisan / tinker
- **定義**: artisan = Laravel 付属のコマンドラインツール(雛形生成 `make:model`、マイグレーション実行、ルート一覧など)。tinker = アプリのコードをその場で実行できる対話シェル。
- **なぜ**: 手書きより速く・規約どおりのファイルが作れる。tinker は「モデルが正しく動くか」を画面を作る前に確かめられる。
- **実例**: `php artisan make:model Todo -m`(モデル+マイグレーション同時生成)。tinker での保存・取得テストは [Sprint 2 レビュー](../02_sprint2/sprint-review.md)。

### マイグレーション
- **定義**: DB のテーブル定義を PHP コードで管理し、コマンドで適用する仕組み。`up()` = 適用、`down()` = 巻き戻し。
- **なぜ**: テーブル定義がコードとして Git に残るので、「誰がいつどんな構造変更をしたか」が追え、他の環境でも同じ DB を再現できる。変更と取り消しをセットで書くのが流儀。
- **実例**: `src/database/migrations/2026_07_02_230759_create_todos_table.php`。`string('title', 100)` が MySQL では `varchar(100)` になる対応は [Sprint 2 レビュー](../02_sprint2/sprint-review.md) 参照。

### Eloquent(ORM)と「設定より規約」
- **定義**: Eloquent = DB のテーブルの行を PHP のオブジェクトとして扱う仕組み(ORM)。SQL を書かずに `Todo::create()` や `Todo::latest()->get()` と書ける。
- **規約**: モデル名は**単数形**(`Todo`)、テーブル名は**複数形**(`todos`)。この命名規約に従うと**設定ゼロで自動的に紐づく**。Laravel は「設定より規約」の思想で、規約に乗ると書くコードが激減する。
- **実例**: `src/app/Models/Todo.php` は最初ほぼ空のままで動いた。[Step 4](dev-walkthrough.md#step-4-todos-テーブルとモデル--91d71ee)

### $fillable とマスアサインメント攻撃
- **定義**: `$fillable` = 「フォーム等からまとめて代入してよいカラム」の許可リスト。
- **なぜ**: 悪意あるユーザーはフォームに**存在しない項目を紛れ込ませて送信できる**(例: `is_admin=1`)。Laravel は「明示的に許可したカラムしか一括代入しない」設計にすることで、この**マスアサインメント攻撃**を防ぐ。`$fillable` がないと `Todo::create($validated)` はエラーになる。
- **実例**: `src/app/Models/Todo.php`

### casts(型変換)
- **定義**: DB の値を PHP の型に変換するルール。
- **なぜ**: MySQL には真偽値型がなく `tinyint(1)` の 0/1 で表現される。casts に `'completed' => 'boolean'` と書くと PHP 側では常に `true/false` で扱える。
- **実例**: `src/app/Models/Todo.php`。tinker で `false` が返ることを確認済み。

### Blade(テンプレート)
- **定義**: HTML の中に `{{ $変数 }}` や `@if` `@foreach` を書ける Laravel のテンプレート機能。`@extends`(親レイアウトの継承)と `@yield` / `@section`(差し込み口と中身)で共通の枠+各ページの中身の 2 層にできる。
- **なぜ**: 共通の HTML 骨格を毎ページ書くのは無駄。また **`{{ }}` は自動で HTML エスケープ**されるので、ユーザー入力を表示してもスクリプトを混入される攻撃(**XSS**)にならない。
- **実例**: `src/resources/views/layout.blade.php`(親)と `todos/*.blade.php`(子)。

### バリデーション
- **定義**: 不正な入力をはじく仕組み。`$request->validate([ルール], [メッセージ])`。
- **なぜ**: 「タイトルが空の TODO」のような不正データを DB に入れない。検証に失敗すると **Laravel が自動で前の画面に差し戻し**、エラー(`$errors`)と入力値(`old()`)を運んでくれる — 自分で if 文とリダイレクトを書く必要はない。
- **境界値**: 「100 文字 OK / 101 文字 NG」まで確認して初めて実装完了と言える。
- **実例**: `TodoController.php` の `validateTodo()`。[Step 6](dev-walkthrough.md#step-6-新規作成us-1--223db9e)

### CSRF 対策(@csrf)
- **定義**: POST 系フォームに `@csrf` と書くと、Laravel が合言葉(トークン)を隠しフィールドとして埋め込み、正しいトークンのない送信を拒否する(419 エラー)。
- **なぜ**: **CSRF 攻撃** = 悪意あるサイトが、ログイン中のあなたのブラウザを騙して勝手にフォームを送信させる攻撃。トークンの照合で「本当に自分のサイトのフォームから送られたか」を確かめる。
- **実例**: すべての POST/PUT/PATCH/DELETE フォームに `@csrf` がある。curl でテストするときも本物のブラウザ同様「①フォームを開いてトークンと Cookie をもらう → ②添えて送信」の手順が必要だった([Sprint 2 レビュー](../02_sprint2/sprint-review.md))。

### PRG パターン(Post → Redirect → Get)
- **定義**: 保存処理(POST)の後は画面を直接返さず、リダイレクト(302)して GET で表示させる定石。
- **なぜ**: 保存直後の画面でブラウザ更新(F5)をすると**同じ POST が再送されて二重登録**になる。間にリダイレクトを挟むと F5 は GET の再実行になるので安全。
- **実例**: `store()` / `update()` / `destroy()` の `return redirect()->route(...)`。

### フラッシュデータ(セッションの一時データ)
- **定義**: 「**次の 1 リクエストだけ**有効」なセッションデータ。成功メッセージ(`with('success', ...)`)、バリデーションエラー、old 値はすべてこれ。
- **なぜ**: リダイレクト先に一度だけメッセージを渡したい、という PRG パターンとセットの仕組み。1 回表示されると消える。
- **落とし穴**: テストで確認の GET を 2 回に分けると 2 回目には消えており、バグと誤認する(実際に起きた: [Sprint 4 レビュー](../04_sprint4/sprint-review.md))。

## Laravel 編(Sprint 3〜4 で登場)

### ルートモデルバインディング
- **定義**: ルートを `/todos/{todo}`、コントローラ引数を `show(Todo $todo)` と**型付き**で書くと、Laravel が URL の ID から自動で DB 検索し、**見つからなければ自動で 404** を返す仕組み。
- **なぜ**: 「ID で検索 → なければ 404」という定型コード(`findOrFail`)を毎回書かなくて済む。フレームワークの機能は「手で書くと面倒な定型処理の自動化」と捉えると学びやすい。
- **実例**: `TodoController.php` の `show()` 以降のメソッド。[Step 7](dev-walkthrough.md#step-7-詳細表示us-3--78b8a8c)

### ルート定義の順序
- **定義**: **具体的なパスが先、可変パスが後**。`/todos/create` は `/todos/{todo}` より前に定義する。
- **なぜ**: 逆にすると "create" という文字列が `{todo}` に飲み込まれ、「ID が create の TODO を探す」と誤解釈される。
- **実例**: `src/routes/web.php` のコメント参照。

### メソッドスプーフィング(@method)
- **定義**: フォームに `@method('PUT')` と書くと隠しフィールド `_method=PUT` が埋め込まれ、Laravel が POST を PUT として読み替える仕組み。
- **なぜ**: REST では更新 = PUT/PATCH、削除 = DELETE が慣習だが、**HTML のフォームは GET と POST しか送れない**。その制約をフレームワーク側で補っている。
- **実例**: `todos/edit.blade.php`(PUT)、`todos/delete.blade.php`(DELETE)、一覧の切替フォーム(PATCH)。

### 設定の 2 段構え(環境変数 → config)
- **定義**: Laravel の設定は「環境変数(.env や実行環境)→ config ファイル → アプリ」の順に流れる。config が `env('APP_TIMEZONE', 'UTC')` のように**参照していなければ、環境変数は無視される**。
- **なぜ知るか**: 「環境変数を設定したのに反映されない」という**エラーの出ない不具合**の典型原因。調査は「環境変数はコンテナに届いているか → config が読んでいるか」と経路の外側から 1 段ずつ確認する。
- **実例**: タイムゾーン修正の 2 コミット(`064f055` → `3273804`)。[Step 8-9](dev-walkthrough.md#step-8-9-タイムゾーン修正--064f055--3273804)

### DRY(Don't Repeat Yourself)
- **定義**: 同じロジックを 2 箇所に書かない原則。
- **なぜ**: 作成と編集のバリデーションをコピペで 2 箇所に持つと、将来「150 文字に変更」のとき片方だけ直して不整合になる事故が起きる。「同じものを 2 回書いたら共通化を考える」。
- **実例**: `TodoController.php` の `validateTodo()`。共通化のリファクタリング差分は [Step 10](dev-walkthrough.md#step-10-編集us-4--cd03e16)。

### 取り返しのつかない操作のワンクッション(確認ページ方式)
- **定義**: 削除など Undo できない操作の前に、専用の確認ページ(GET)を挟み、実行(DELETE)と分離する設計。
- **なぜ**: 誤クリックでデータが消えるのを防ぐ。JS の `confirm()` に頼らなくても、「確認を見る = GET」「実行する = POST 系」の分離だけで安全にできる。UI の安全性は実装テクニックではなく**設計段階**で決まる。
- **実例**: `todos/delete.blade.php`。設計図は [Sprint 4 バックログ](../04_sprint4/sprint-backlog.md)。
