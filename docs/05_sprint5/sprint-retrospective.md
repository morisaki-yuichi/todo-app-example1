## 学びの要約

### 学び1: コンテナとアプリが分離されていれば、環境は「載せ替えられる」
bitnami から Sail へ土台を丸ごと入れ替えても、コントローラ・ビュー・マイグレーションは 1 行も変わらなかった。マイグレーションのおかげで DB も新しい環境でコマンド一発で再現できた。「環境をコードで管理する」ことの価値が一番よく分かった瞬間だった。

### 学び2: ツールの生成物は git status で棚卸しする
sail:install は compose.yaml だけでなく .env や phpunit.xml まで書き換えていた。ファイル名も docker-compose.yml ではなく compose.yaml で、思い込みで探すと「生成されていない」と誤認する。ツールを実行したら git status で「何が変わったか」を必ず確認する。

### 学び3: 同じ問題でも、構成が良いと対処が軽くなる
ポート衝突は Sprint 1 でも起きたが、あのときは compose ファイルの編集が必要だった。今回は .env に 1 行(VITE_PORT=5273)で済んだ。「設定を外に出す」設計は、トラブル対応のコストまで下げる。

### 学び4: 移行作業は「退路の確保」から始める
旧環境の停止は down(ボリュームは残る)にとどめ、いつでも戻れる状態で進めた。新環境での全機能確認が終わるまで、旧データは消さない。
