## 学びの要約(開発者自身の言葉で)

### 学び1: LaravelのMVCは「一本道」で覚える
リクエストは「URL(ルーティング)→ コントローラ → モデル ⇔ DB → ビュー → HTML」の順に流れる。どこかでエラーが出たら、この道のどこで止まったかを考えると調査場所を絞れる。

### 学び2: 規約に従うと設定がいらない
モデルを `Todo`(単数形)にすると、テーブル `todos`(複数形)が自動で紐づいた。Laravelは「設定より規約」の思想で、命名規約に乗ると書くコードが激減する。逆に規約を外れると自分で設定を書くことになる。

### 学び3: フォーム1機能にはルートが2つ要る
「フォームを見せるGET」と「送信を受けるPOST」は別のリクエスト。POSTには `@csrf` が必須で、書き忘れると419エラーになる。保存後は画面を返さずリダイレクトする(PRGパターン)ことで、F5による二重登録を防ぐ。

### 学び4: バリデーションは「差し戻し」までがワンセット
`$request->validate()` は失敗すると自動で前の画面に戻り、エラー(`$errors`)と入力値(`old()`)を運んでくれる。自分で if 文を書いてリダイレクトする必要はない。境界値(100文字OK/101文字NG)まで確認して初めて「バリデーションを実装した」と言える。

## KPT

### Keep(良かったこと・続けたいこと)
- 前スプリントのTry「マージ後の動作確認までがマージ作業」を実践し、今回はマージ後の500エラーを未然に防げた
- 「動く状態でしかコミットしない」を守るため、未実装ルートへのリンクをコミットから外す判断ができた
- 正常系だけでなく異常系・境界値まで確認してからタスク完了とした

### Problem(困ったこと・うまくいかなかったこと)
- curl でのフォーム送信で CSRF トークンと Cookie の扱いに手間取った(ブラウザが裏でやってくれていることを初めて意識した)
- 新しい概念(fillable、casts、PRG など)が一度に多く登場し、消化に時間がかかった

### Try(次のスプリントで試したいこと)
- Sprint 3 の編集機能は今回の作成機能と構造が似ているはず。「まず自分ならこう書く」と予想してから説明を聞く
- ルート定義は「具体的なパス(/todos/create)を可変パス(/todos/{id})より先に書く」を意識する(Sprint 3 で詳細表示を作るときの注意点として先取り)
